# Phase 9.1: Code Review Report

**Reviewed:** 2026-05-18T00:00:00Z
**Depth:** standard
**Files Reviewed:** 6
**Status:** issues_found

## Summary

This phase wires `step_duration_ms` from the `_log_step` timing closure into `ActionDetailEvent` and renders it in the narration feed as a seconds ticker. The core implementation is functionally correct — the field is declared in `events.py`, populated in `runner.py`'s `_log_step`, and consumed in `index.html`'s `handleActionDetail`. The test suite covers the field contract thoroughly.

However, one critical bug exists: `step_start` is initialised before `pre_flight_check` runs, meaning that the pre-flight network I/O (Ollama tag fetch, Anthropic/OpenAI model-list call — potentially several seconds) is silently baked into `duration_ms` of the very first step, making the first step's timing meaningless. Four additional warnings are documented below.

---

## Critical Issues

### CR-01: `step_start` is captured before `pre_flight_check`, inflating first-step duration

**File:** `agent/runner.py:476`

**Issue:** `step_start = time.monotonic()` is assigned at line 476, before the `pre_flight_check` call at line 499 (which performs network I/O — an HTTP call to Ollama's `/api/tags` endpoint, or a live Anthropic/OpenAI models-list request). Pre-flight can take 0.1–5 s depending on network conditions. That entire interval is then counted as "step 1 elapsed time" when `_log_step` computes `duration_ms = int((time.monotonic() - step_start) * 1000)` at line 541. The `step_start` timer should not start until the agent actually begins executing steps.

**Impact:** Every run's first `ActionDetailEvent.step_duration_ms` and first JSONL record's `step_duration_ms` field are overstated by the pre-flight duration. The UI timestamp badge for step 1 is therefore wrong. Training data that consumes `step_duration_ms` as a step-performance signal is corrupted for step 1.

**Fix:** Move `step_start = time.monotonic()` to just before `agent.run(...)` is called (line ~607), inside the `else` branch after `pre_flight_check` succeeds and the browser/agent are set up. A clean location is immediately before the `await asyncio.wait_for(agent.run(...))` call:

```python
# Reset the step timer immediately before the agent loop starts,
# not at the top of run_agent where pre_flight I/O contaminates step 1.
step_start = time.monotonic()
try:
    history = await asyncio.wait_for(
        agent.run(max_steps=config.max_steps, on_step_end=_log_step),
        timeout=config.session_timeout,
    )
```

Remove the line `step_start = time.monotonic()` from line 476, keeping the `nonlocal step_start` declaration in `_log_step` unchanged. The `step_start` variable can still be initialised at line 476 as a sentinel (e.g., `step_start: float = 0.0`) so the `nonlocal` reference is valid, but it must be reset to `time.monotonic()` immediately before `agent.run`.

## Warnings

### WR-01: Badge-detection heuristic in `handleActionDetail` is brittle and can misclassify

**File:** `agent/templates/index.html:432-433`

**Issue:** The idempotency guard for action badges checks `first.className.indexOf('action-badge-') === 0`, meaning it only recognises a badge if it is literally the first child of the row. However, `handleThought` may have already appended a `<details class="thought-details">` element to the row before `handleActionDetail` fires (SSE events are not strictly ordered between `sse-swap` targets). If `thought_details` arrives first, `first.className` will be `'thought-details'`, not an `'action-badge-*'` class, so the guard fires `false` and a duplicate badge is inserted.

In practice, `ThoughtEvent` fires before `ActionDetailEvent` (enforced by the pre-step callback ordering), and `_ensureNarrationRow` creates the row with a `step-num` span as its first child. After that, `handleThought` appends a `<details>` — making it the second child. If `handleActionDetail` then reads `row.firstChild`, it sees the `.step-num` span (not a badge), and correctly inserts the badge before it. But this ordering is not enforced by the DOM: if the network delivers the two SSE events in a different order (entirely possible since they are dispatched on separate `sse-swap` bridge elements), the guard logic breaks.

**Fix:** Use `row.querySelector('[class^="action-badge-"]')` instead of inspecting `firstChild`:

```javascript
const alreadyBadged = !!row.querySelector('[class^="action-badge-"]');
```

This is order-independent and survives any child-node ordering.

### WR-02: `_buildActionBadge` maps `'extract'` to `action-badge-scroll` (copy-paste error)

**File:** `agent/templates/index.html:347`

**Issue:** In `_buildActionBadge`, the `extract` action type is mapped to `'action-badge-scroll'` — the same class as `scroll`. This is almost certainly a copy-paste error; `extract` is a distinct action (page content extraction) and should have a distinct visual badge, or at minimum a semantically correct fallback. Rendering it as a scroll badge produces a misleading UI label.

```javascript
const map = {
    navigate: 'action-badge-navigate',
    click:    'action-badge-click',
    type:     'action-badge-type',
    scroll:   'action-badge-scroll',
    extract:  'action-badge-scroll',   // <-- wrong class
    done:     'action-badge-click',
};
```

**Fix:** Add a distinct `action-badge-extract` CSS class, or map to the generic fallback explicitly with a comment:

```javascript
extract:  'action-badge-extract',   // add .action-badge-extract to style.css
// OR, if no distinct style is desired yet:
extract:  'action-badge-default',   // document the intentional fallback
```

At minimum, remove the misleading `action-badge-scroll` association so the fallback (`'action-badge-scroll'` from the `|| 'action-badge-scroll'` default) at line 351 applies, making the mapping table correct by omission.

### WR-03: `thought` body node has `aria-hidden="true"` set initially, hiding content from screen readers even when `<details>` is open

**File:** `agent/templates/index.html:396`

**Issue:** In `handleThought`, the body `<div>` inside `<details>` is initialised with `aria-hidden="true"` at line 396. When a user opens the `<details>` element, the `toggle` event listener at line 418 updates `aria-hidden` to `"false"`. However, browser `<details>/<summary>` semantics already hide collapsed content from the accessibility tree natively — adding an explicit `aria-hidden` layer creates a timing window where a screen reader may announce the open state before the `toggle` listener fires, or may be permanently confused by the attribute on interactive content.

More critically, the initial `aria-hidden="true"` on the body div means that if JavaScript fails to register the `toggle` listener (e.g., a future Alpine/HTMX upgrade changes event ordering), the content would be permanently hidden from assistive technology even when visually expanded.

**Fix:** Remove the initial `aria-hidden` attribute entirely and rely on the native `<details>` accessibility semantics:

```javascript
// Remove: body.setAttribute('aria-hidden', 'true');
// Remove the toggle listener that flips aria-hidden
details.appendChild(body);
// The native <details> element already hides collapsed content from a11y tree
```

If an explicit ARIA attribute is desired for progressive enhancement, set it only inside the `toggle` handler (not as an initialisation step):

```javascript
details.addEventListener('toggle', () => {
    body.setAttribute('aria-hidden', String(!details.open));
});
// Do NOT set it before appending to DOM
```

### WR-04: Integration test `test_log_step_puts_duration_on_action_detail_event` asserts `>= 0` instead of `> 0`, allowing a zero-ms duration to pass silently

**File:** `tests/unit/test_events_phase9_1.py:103`

**Issue:** The integration test for PERF-01 asserts `action_detail_events[0].step_duration_ms >= 0`. This assertion is too weak — it allows `0` as a valid duration. A zero value would indicate the timer was either not started, or `step_start` was reassigned to `time.monotonic()` immediately before `_log_step` ran (i.e., bug CR-01 in its most pathological form). The test sleeps 10 ms (`await asyncio.sleep(0.010)`) to ensure measurable elapsed time, so the assertion should enforce `> 0`:

```python
assert action_detail_events[0].step_duration_ms > 0, (
    f"step_duration_ms must be positive (> 0); got {action_detail_events[0].step_duration_ms}"
)
```

The current assertion `>= 0` would pass even if the implementation returned `0`, masking a broken timing path.

## Info

### IN-01: `_log_step` duplicates action-extraction logic from `log_step`

**File:** `agent/runner.py:558-569`

**Issue:** The action-type and params extraction block inside `_log_step` (lines 558–569) is a near-duplicate of the same logic inside `log_step` (lines 369–376). Both closures independently call `agent.history.model_actions()`, extract `action_type` via `next(k for k in last_action if k != "interacted_element", "unknown")`, and extract `params`. This duplication means a future change to extraction logic must be applied in two places.

**Fix:** Extract a shared helper `_extract_action_fields(agent_instance) -> dict` and call it from both `log_step` and `_log_step`. This is a refactor, not a correctness issue.

### IN-02: Phase 9.1 test file missing `@pytest.mark.anyio` decorator on async tests

**File:** `tests/unit/test_events_phase9_1.py:71`

**Issue:** `test_log_step_puts_duration_on_action_detail_event` is declared `async def` without a `@pytest.mark.anyio` (or equivalent) decorator. The existing Phase 5 and Phase 6 test files also lack this decorator, which suggests the project uses a `conftest.py` fixture or `pytest-anyio`/`pytest-asyncio` auto-mode setting. This is noted because if the pytest-asyncio `asyncio_mode` is ever changed from `auto` to `strict`, this test will silently be collected but never awaited, producing a vacuous pass. Confirm `asyncio_mode = "auto"` is pinned in `pyproject.toml` or `pytest.ini` to make the intent explicit.
